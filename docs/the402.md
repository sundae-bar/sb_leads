# the402.ai integration — the managed marketplace path

## What this is, in plain English

**the402.ai** is a marketplace for AI services. Think of it like Stripe + a catalog page — buyers (other AI agents) browse listings, pay for a single API call in USDC, and the402 takes a 5% cut. We're listed as **"Scoop"** (service id `svc_b5eb7e39341e4711`) at $0.25 per call.

We never see the buyer. the402 takes their money, then sends *us* a webhook saying "hey, someone bought Scoop with this LinkedIn URL — please fulfil." We run the email lookup, POST the results back to the402's callback URL, and they handle delivering it to the buyer. Cleared funds show up in our linked wallet within their settlement window.

## Why it matters

The agent economy needs a way for an AI to **discover, pay for, and use** a service in a single transaction — no signup, no API keys to manage, no waiting for an invoice. the402.ai is one of two marketplaces solving that (the other is the x402 ecosystem, see [x402.md](./x402.md)). Different audiences, same `findEmails()` engine.

This was the easier ramp because the402 manages the buyer side — billing, dispute handling, the buyer's UX. We just need to (a) list the service, (b) accept a webhook, (c) reply with the deliverable.

## What's in the code

| File | Purpose |
|---|---|
| [services.ts](../apps/api/src/integrations/the402/services.ts) | **Declarative manifest** — defines `LISTED_SERVICES` (just `Scoop` for now). The single source of truth for what we offer, price, input/deliverable schemas. |
| [signature.ts](../apps/api/src/integrations/the402/signature.ts) | HMAC-SHA256 verifier for incoming webhooks. Signs over `${X-Webhook-Timestamp}.${rawBody}`. Rejects timestamps older than 5 minutes (replay protection). |
| [webhook.ts](../apps/api/src/integrations/the402/webhook.ts) | The Express handler. Verifies HMAC → idempotency check via the `the402_events` table → dispatches by `service_name` → calls `findEmails()` → POSTs the deliverable to `event.callback_url` → marks `processed_at`. **Logs** `the402 job delivered` on success and a stack trace on failure. |
| [sync-the402-services.ts](../apps/api/scripts/sync-the402-services.ts) | One-shot script (`pnpm the402:sync`) that POSTs our manifest to `/v1/services`. Use it after any change to `services.ts`. |
| Migration [0015_the402_events.sql](../supabase/migrations/0015_the402_events.sql) | The `the402_events` table — event audit + idempotency keys. No RLS; internal only. |

## What we expect from the402, and what we return

### Inbound: `POST /the402/webhook`

The402 sends events shaped like:

```json
{
  "id": "evt_…",
  "type": "job_dispatch",                     // also: thread_inquiry, quote_request (we 200-ack the latter two)
  "service_name": "Scoop",                    // or service_id
  "callback_url": "https://api.the402.ai/v1/jobs/<id>/callback",
  "inputs": {
    "linkedin_url": "https://www.linkedin.com/in/<handle>",
    "email_types": ["work", "personal"],      // optional
    "verify": false                           // optional
  }
}
```

Headers:
- `X-Webhook-Timestamp: <unix-seconds>`
- `X-Webhook-Signature: sha256=<hex>`

The signature is HMAC-SHA256 of `${timestamp}.${rawBody}` with our shared `THE402_WEBHOOK_SECRET`. If it fails verification we 401 immediately and write nothing.

### Outbound: deliverable to the callback URL

```json
{
  "deliverable": {
    "linkedin_url": "https://…",
    "emails": [
      {
        "address": "satya@microsoft.com",
        "type": "work",
        "verified": true,
        "source_provider": "apollo",
        "confidence": 0.92
      }
    ],
    "person": { "full_name": "Satya Nadella", "title": "Chairman & CEO", … },
    "company": { "name": "Microsoft", "domain": "microsoft.com" },
    "providers_attempted": [
      { "provider": "aleads", "found": false },
      { "provider": "apollo", "found": true }
    ]
  }
}
```

We send this with `X-API-Key: $THE402_API_KEY` so the402 can attribute it to our provider account.

### Refund-on-empty

We **call `consumeCredits` upfront** but then **refund** if `findEmails()` returns zero emails — same semantics as the REST and chat paths. The buyer never gets billed for misses. See [findEmail.ts:81-97](../apps/api/src/services/findEmail.ts) and [webhook.ts:144-152](../apps/api/src/integrations/the402/webhook.ts).

> Note: the *user-facing* "credits" don't apply on this path (the402 collected USDC, not our credits). The refund here is symbolic — we run `consumeCredits(adminTenantId, 1)` then `refundCredits` so the audit trail records the attempt. In practice the402 is on the hook for what the buyer paid.

## Env vars

```
THE402_API_URL=https://api.the402.ai
THE402_API_KEY=sk_…                            # used for sync + earnings + outbound deliverable POSTs
THE402_WEBHOOK_SECRET=whsec_…                  # HMAC key for inbound webhooks
```

Both keys are issued by the402 when you register as a provider. They live in `.env` locally and on Railway.

## Operational walkthroughs

### Register / re-register the service

```bash
pnpm --filter @scoop/api the402:sync
```

The script reads `LISTED_SERVICES` from [services.ts](../apps/api/src/integrations/the402/services.ts) and `POST`s each to `/v1/services`. Idempotent — re-runs after a manifest change just no-op for unchanged services. Find the resulting `svc_…` id in the the402 dashboard.

### Check earnings

Two places:

1. The super-admin dashboard route at `https://scoop-web-production.up.railway.app/app/admin/the402` — server-side fetch of `/v1/provider/earnings`, shows settled / held / pending.
2. CLI:
   ```bash
   curl -H "X-API-Key: $THE402_API_KEY" https://api.the402.ai/v1/provider/earnings | jq
   ```

### Trigger a test job from the402

In the402 dashboard, go to **Services → Scoop → Test**. You'll send a synthetic `job_dispatch`. Tail Railway logs (`api` service) for:

- `POST /the402/webhook` at 200
- `the402 job delivered { event_id: ..., emails_found: N }`
- A `the402_events` row in Supabase with `processed_at` set and `error` null

### Debug a failed job

Every event lands in `the402_events`. Query the most recent failures:

```sql
SELECT event_id, event_type, error, payload
FROM the402_events
WHERE error IS NOT NULL
ORDER BY created_at DESC LIMIT 20;
```

The `error` column captures any exception message from the handler. The `payload` column has the full inbound event for replay.

## Known constraints

- **One service per dashboard listing.** the402 ties a `service_name` to a `svc_…` id; renaming requires a delete + re-create. Our handler matches by `service_name`, so if you rename in [services.ts](../apps/api/src/integrations/the402/services.ts), buy back via the dashboard or update the listing's name via `PATCH /v1/services/<id>` *before* the next sync.
- **No partial settlements.** the402 takes the full price regardless. The refund logic above is purely on our internal credit accounting; the402's $0.25-minus-5% goes through whether we found an email or not.
- **5% platform fee.** Net to us per call: $0.2375. Factored into pricing.
