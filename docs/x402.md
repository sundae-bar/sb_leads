# x402 integration — the raw on-chain payment path

## What this is, in plain English

**x402** is an open payment protocol that resurrects HTTP's long-forgotten `402 Payment Required` status code. It works like a vending machine: an agent makes an HTTP request, our server responds with "you need to pay $0.25 to access this," the agent signs a USDC payment authorization and retries, and our server delivers.

There's **no signup, no API key, no Bearer token**. The payment authorization itself is the auth. The buyer (any AI agent on Base mainnet with USDC) finds us, pays us directly, gets the result.

This is a *separate path* from the402.ai's managed marketplace. the402 wraps x402 in a UI + dispute system; the x402 protocol itself is the raw rails. We're on both.

## Why it matters

Two reasons:

1. **Discoverability**. Coinbase runs a service catalog called **Bazaar** that crawls x402-enabled endpoints. Anything in Bazaar appears on **agentic.market** (the public-facing UI on top of Bazaar) and any other x402 ecosystem directory. The audience is "every AI agent with an x402 client" — currently smaller than the402's but growing fast.
2. **No fees**. Unlike the402's 5%, x402 protocol itself is free. Coinbase's facilitator (the trusted third party that verifies payments) is free for 1,000 calls/month then $0.001/call.

The economics: at $0.25/call, the402 nets us $0.2375 and x402 nets us $0.249. The cost difference is small at this volume; the bigger lever is *which audience pays first*.

## The two scheme question — why "exact" and not "upto"

x402 has two payment schemes:

- **`exact`** — buyer signs for *exactly* $0.25, facilitator rejects anything less. All-or-nothing.
- **`upto`** — buyer signs for *up to* $0.25, server can settle partial / zero amounts. Refunds possible.

We use `exact`. We tried `upto` for refund-on-empty parity with the other paths, but it requires the buyer to send a separate Permit2 approval transaction *first*, which costs them gas and adds latency. Most x402 clients in the wild don't implement that flow. We chose compatibility over refunds.

**Consequence**: `exact` has no in-protocol refund. Rather than charge-and-keep on a miss, we never *settle* a failed or empty lookup in the first place — see [Payment lifecycle & error contract](#payment-lifecycle--error-contract) below. (The other paths — dashboard credits, the402 — refund after charging; x402 simply doesn't charge.)

## Payment lifecycle & error contract

The x402 flow is **verify → work → settle**. `verify` is an off-chain signature
check — *no funds move*. Only `settle` submits the on-chain
`transferWithAuthorization`. The `@x402/express` middleware buffers our
handler's response and settles **only on a 2xx**; on any **>= 400 it cancels**
(no funds move). So the handler's HTTP status *is* the billing decision, and we
use that to never charge for a failed or empty result:

| Outcome | Status | Settles? | `code` |
|---|---|---|---|
| ≥1 email found | `200` | ✅ charged | — (deliverable, `charged:true`) |
| zero emails found | `404` | ❌ | `no_email_found` |
| provider fan-out timed out | `504` | ❌ | `provider_timeout` (`retryable:true`) |
| duplicate of a paid request | `200` | ❌ settle skipped by hook | — (cached deliverable, `charged:false`, `duplicate:true`) |
| bad input | `400` | ❌ | `invalid_input` |
| unexpected failure | `502`/`500` | ❌ | `provider_error` / `internal_error` |

Every non-2xx carries a structured body so a client never has to guess whether
it paid:

```json
{
  "error": { "code": "no_email_found", "message": "No email found for the given input" },
  "charged": false,
  "settlementTx": null,
  "requestId": "…",
  "retryable": false
}
```

Implemented in [routes/x402-find-email.ts](../apps/api/src/routes/x402-find-email.ts)
+ [integrations/x402/responses.ts](../apps/api/src/integrations/x402/responses.ts).

**Timeout budget.** `findEmails` is wrapped in `X402_FIND_EMAIL_TIMEOUT_MS`
(default 20s). On expiry we return `504` → cancel, so a slow/hung provider can't
make the request quietly succeed and settle *after* the client has already timed
out. The deadline also aborts the lookup itself (an `AbortSignal` threaded
through `findEmails` into every provider's HTTP calls): the waterfall stops
before its next round, verification is skipped, and in-flight provider requests
are cancelled — a timed-out request stops spending provider credits.

**Idempotency (retry double-charge).** A buyer's retries sign a *fresh* nonce
each time, so we key dedup on **who + what** — an `Idempotency-Key` header if
present, else `hash(payer_address + normalized_request)` — not the nonce. Two
parts ([idempotency.ts](../apps/api/src/integrations/x402/idempotency.ts) +
[settlement.ts](../apps/api/src/integrations/x402/settlement.ts)):

- The **`onBeforeSettle` hook** is the authoritative guard. Right before funds
  move, if it has already settled this `(payer + request)` it returns
  `{ skip: true, result }` — the buyer still gets a normal `200` + data, but no
  second charge. `onSettleFailure` un-marks the key so a genuinely-failed
  payment's retry is charged properly.
- The **handler cache** serves the cached deliverable so a retry doesn't
  re-spend providers, and reports `charged:false` / `duplicate:true` on a
  confirmed replay.
- The **in-flight lock** makes genuinely-*concurrent* duplicates join the
  already-running lookup instead of each starting their own provider waterfall
  (otherwise N simultaneous copies of one request — each a freshly-signed
  authorization from one funded wallet — would cost us N provider runs for a
  single 0.25 payment). All joined requests share one outcome, success or
  failure.

So a retry storm for one lookup collapses to a single charge **and** a single
provider spend, **without client cooperation**. **Limitations:** scoped per
payer (a *different* wallet or changed input is correctly a new charge);
in-process only (fine for the single replica, lost on redeploy, not shared
across instances — swap the `idempotency` store for a durable one when
scaling). Under concurrency the body's `charged` flag is best-effort (the
original's settlement may still be in flight when a joined response is
written); `X-PAYMENT-RESPONSE` (empty tx = not settled) is authoritative.

**Settlement latency.** On the 200 path the middleware awaits on-chain
settlement *before* returning the body, so a successful call can take a few
seconds. Clients should set a request timeout of ≥30s; the settlement tx hash
comes back in the `X-PAYMENT-RESPONSE` header.

## Env vars

```
# x402 protocol config
X402_FACILITATOR_URL=https://api.cdp.coinbase.com/platform/v2/x402    # CDP mainnet
X402_NETWORK=eip155:8453                                              # Base mainnet (testnet: eip155:84532)
X402_PAY_TO_ADDRESS=0x48E23742181654139F9A7c1b14dd64B04e89ed3b        # our CDP-managed EOA
X402_FIND_EMAIL_PRICE=$0.25
X402_DISCOVERABLE=true                                                # turns on Bazaar crawling — mainnet only

# Coinbase Developer Platform (for CDP facilitator auth + wallet management)
CDP_API_KEY_ID=…
CDP_API_KEY_SECRET=…                                                  # facilitator auth-header signing
CDP_WALLET_SECRET=…                                                   # programmatic wallet access (creation, signing)
```

The CDP facilitator requires Ed25519-signed JWT headers on every request. The `@coinbase/x402` package's `createFacilitatorConfig(id, secret)` handles the signing; we use it conditionally in [server.ts](../apps/api/src/integrations/x402/server.ts) so the codepath also works against the unauthenticated testnet facilitator.

## How discoverability works

```
our /x402/find-email endpoint (with X402_DISCOVERABLE=true)
       │
       │  declares  extensions.bazaar.discoverable: true
       │  in every 402 response
       ▼
  Coinbase Bazaar crawler
  - GETs https://api.scoop.sundaebar.ai/x402/find-email
    on a schedule
  - sees the bazaar block in the PAYMENT-REQUIRED header
  - indexes us in api.cdp.coinbase.com/platform/v2/x402/discovery/resources
       │
       ▼
  agentic.market
  - polls Bazaar's discovery feed nightly
  - mirrors entries to api.agentic.market/v1/services
  - renders them on the public UI at https://agentic.market
       │
       ▼
  any AI agent searching for "email" / "linkedin" / "enrichment"
```

Bazaar typically picks us up within minutes once `X402_DISCOVERABLE=true`. agentic.market lags by hours-to-a-day depending on their cron. Use `pnpm discovery:poll` to watch both.

### Listing metadata (name / tags / icon)

The catalog's display name, tag chips, and icon come from three service-level
fields on the route's `resource` (the x402 `ResourceInfo`), set in
[server.ts](../apps/api/src/integrations/x402/server.ts):

| Route field | Catalog field | Constraint |
|---|---|---|
| `serviceName` | listing name (else falls back to the bare host, e.g. `api.scoop.sundaebar.ai`) | ≤32 printable-ASCII chars |
| `tags` | tag chips | ≤5 tags, each ≤32 ASCII chars |
| `iconUrl` | listing icon | ≤2048 chars; must be publicly reachable |

These are validated by the resource server's `ResourceInfoSchema` — an
over-length `serviceName` or a 6th tag is silently dropped. They require
`@x402/* >= 2.14.0`; on 2.12.x the express middleware built `ResourceInfo`
from only `url`/`description`/`mimeType` and dropped these three fields.
agentic.market also runs its own enrichment pass (`provider`, `providerUrl`,
`category`, top-level `name`) that we can't set through x402 metadata.

## Operational walkthroughs

### Test the full payment cycle on testnet

```bash
# Local API on testnet
pnpm --filter @scoop/api dev

# In another terminal
pnpm --filter @scoop/api x402:self-test
```

The script generates a throwaway buyer wallet, requests Sepolia ETH + USDC from the CDP faucet, signs the EIP-3009 payment, hits our local endpoint. Expect HTTP 200 with the deliverable in ~5 seconds.

### Test against production

```bash
X402_TEST_ENDPOINT=https://api.scoop.sundaebar.ai/x402/find-email \
  pnpm --filter @scoop/api x402:self-test
```

Same script, but the buyer pays *real* USDC (still on Base Sepolia if Railway's `X402_NETWORK` is set to that — verify with the `discovery:poll once` output or by curling `/x402/find-email` unsigned and decoding the `PAYMENT-REQUIRED` header).

> 🚨 If Railway is on mainnet (`eip155:8453`), the buyer in the script needs real Base mainnet USDC — the CDP faucet doesn't fund mainnet. Manually fund the buyer wallet from a real exchange before running, or send testnet test runs only.

### See if we're indexed

```bash
pnpm --filter @scoop/api discovery:poll      # loops every 60s
pnpm --filter @scoop/api discovery:poll once # single check
```

Prints whether Scoop appears in Bazaar + agentic.market.

### Inspect the live 402 response

```bash
curl -s -D /tmp/h.txt -X POST https://api.scoop.sundaebar.ai/x402/find-email \
  -H 'Content-Type: application/json' \
  -d '{"linkedin_url":"https://www.linkedin.com/in/satyanadella/"}'

grep -i "^PAYMENT-REQUIRED:" /tmp/h.txt | sed 's/^[^:]*: //' | base64 -d | jq
```

The decoded body shows the network, asset, amount, payTo, and Bazaar extensions. Useful for confirming env-var flips landed.

## Known constraints

- **No refund on empty — but no charge either.** `exact` can't refund in-protocol, so we don't *settle* empty/failed lookups at all (return >= 400 → cancel). See [Payment lifecycle & error contract](#payment-lifecycle--error-contract).
- **Double-charge/double-spend dedup is per-payer + in-process.** Caught server-side via `hash(payer + request)` (no client cooperation needed) — the settle hook blocks repeat charges and the in-flight lock blocks repeat provider runs. A *different* wallet or changed input is correctly a new charge, and none of it survives a redeploy or spans replicas. See the idempotency note above.
- **`@x402/*` + `@coinbase/x402` are pinned exactly (no caret).** The settlement dedup hook ([settlement.ts](../apps/api/src/integrations/x402/settlement.ts)) reads `@x402/express`'s transport-context shape, which isn't a public contract. A silent minor-version bump could null the dedup key and quietly disable the double-charge guard (it fails open). Upgrade deliberately: bump the pins, run the x402 test suite, and re-run `pnpm x402:self-test` against staging.
- **Rate limiting depends on `TRUST_PROXY_HOPS` being exact.** The limiter keys on `req.ip`, derived from `X-Forwarded-For` via the hop count (Railway edge = 1; Cloudflare-proxied custom domain = 2). Overstate it and clients can spoof their IP past the limiter; understate it and all traffic collapses into the proxy's IP (collective throttling). Verify after any infra change: `curl -H "X-Forwarded-For: 1.2.3.4" https://<api>/health` and check the logged ip.
- **One endpoint per service.** `/x402/find-email` is the only route protected by the middleware right now. Adding more (e.g. `/x402/verify-email`) means adding more entries to the `paymentMiddleware` config — see [extending.md](./extending.md).
- **Facilitator dependency.** If CDP's facilitator is down, all x402 traffic stalls. Health-check by polling our `/health` endpoint *and* CDP's facilitator URL.
- **CDP fee.** First 1,000 txns/month free, then $0.001 each. Negligible at current volume.
